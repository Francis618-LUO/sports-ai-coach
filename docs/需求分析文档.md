# 双运动AI姿态教练 —— 项目需求分析文档

---

## 一、背景意义

### 1.1 项目背景

随着全民健身意识的提升，网球和高尔夫两项运动在中国正迅速普及。然而，这两项运动都属于**高技巧门槛运动**——动作的规范性直接影响击球效果和运动安全。传统模式下，初学者依赖教练一对一指导，费用高昂（网球私教约300-500元/小时，高尔夫教练约800-1500元/小时），且教练资源分布不均，难以随时随地进行动作指导。

与此同时，人工智能技术在计算机视觉领域取得重大突破。以Google MediaPipe为代表的姿态估计算法已能在浏览器端实时提取人体33个关键点，结合国产大语言模型（如DeepSeek）的强大推理能力，使得**低成本、普惠化的AI运动教学**成为可能。

### 1.2 项目意义

1. **降低运动学习门槛**：使用者只需手机拍照即可获得专业级动作分析，无需昂贵私教，普惠运动爱好者。
2. **覆盖两项运动**：同时支持网球和高尔夫，满足项目负责人（本人）真实的双运动需求，区别于市面单一运动分析工具。
3. **AI技术落地教学场景**：将姿态识别与大模型智能分析结合，探索AI在体育教学中的创新应用。
4. **服务真实生活需求**：解决"打球动作不标准却无从纠正"的痛点，具有明确的实用价值。

## 二、可行性分析

### 2.1 技术可行性

| 技术模块 | 选型方案 | 成熟度 |
|----------|----------|--------|
| 姿态关键点检测 | MediaPipe Pose（Google开源） | 成熟，已在生产环境广泛使用 |
| 智能分析与建议生成 | DeepSeek API（国产大语言模型） | 成熟，具备强推理能力，国内直连 |
| 前端框架 | React + TypeScript + Vite | 成熟，生态完善 |
| 响应式设计 | CSS Media Queries + Tailwind CSS | 成熟 |
| 部署 | Vercel（免费托管） | 成熟，一键部署 |

**技术风险低**：核心依赖均为开源且文档完善的成熟技术，无自研算法门槛。MediaPipe Pose的浏览器端推理可在主流手机浏览器上流畅运行，DeepSeek API国内直连、调用稳定。

### 2.2 网络环境与API可达性

本项目部署于中国大陆境内，开发者与目标用户均无VPN。基于此约束，技术选型遵循"全链路国内直连"原则：

- **前端部署**：Vercel在国内可通过边缘节点访问；备选方案为阿里云OSS + CDN或腾讯云静态托管
- **AI API**：选用DeepSeek API（深度求索），服务器位于国内，无需VPN即可调用，响应延迟低
- **姿态检测模型**：MediaPipe Pose模型文件托管于CDN，加载一次后浏览器缓存，后续离线可用
- 本方案在当前网络环境下完全可行。

### 2.3 经济可行性

- **开发成本**：使用免费开源工具，无软件授权费用
- **API费用**：DeepSeek API按token计费，单次动作分析约消耗500-800 tokens（约¥0.002-0.003），作业演示期间费用可忽略不计
- **部署成本**：Vercel免费套餐足够支撑课堂演示
- **总结**：零经济负担，完全可承担

### 2.4 操作可行性

- 用户仅需手机浏览器访问网页、允许摄像头权限、拍照上传即可完成全流程
- 界面设计遵循移动端直觉交互，无需学习成本
- 无需安装任何App，扫码即用

### 2.5 时间可行性

- 开发周期：8天（6月11日—6月19日），预留2天缓冲至6月21日截止日期
- 功能范围已做裁剪，聚焦核心流程：拍照→姿态检测→AI分析→结果展示

## 三、项目需求分析

### 3.1 功能需求

#### 核心功能（Must Have）

| 编号 | 功能 | 描述 |
|------|------|------|
| F1 | 运动类型选择 | 用户选择"网球"或"高尔夫"运动项目 |
| F2 | 动作类型选择 | 网球：正手准备、发球托举、半西方式握拍；高尔夫：站位准备、上杆顶点、推杆姿势 |
| F3 | 拍照/上传 | 调用手机摄像头拍照或从相册选择照片 |
| F4 | 姿态关键点检测 | 浏览器端使用MediaPipe Pose提取人体33个关键点坐标 |
| F5 | 骨骼线可视化 | 在原图上叠加骨骼连接线和关键点标注 |
| F6 | 关节角度计算 | 提取肩、肘、髋、膝等关节角度数据 |
| F7 | AI动作分析 | 将姿态数据发送至DeepSeek API，生成动作规范性评价 |
| F8 | 训练建议 | AI给出分步骤的改进训练方案 |
| F9 | 结果展示页 | 展示标注图、角度数据、AI分析报告、训练建议 |

#### 扩展功能（Nice to Have）

| 编号 | 功能 | 描述 |
|------|------|------|
| F10 | 标准动作对比 | 并排展示用户动作与标准动作参考图 |
| F11 | 历史记录 | 保存分析历史，追踪动作改善趋势 |
| F12 | 语音播报 | AI分析结果语音朗读 |

### 3.2 非功能需求

| 需求 | 指标 |
|------|------|
| 响应时间 | 姿态检测 < 2秒，AI分析 < 5秒 |
| 兼容性 | iOS Safari、Android Chrome 主流版本 |
| 界面适配 | 手机端优先，同时适配平板和桌面端 |
| 隐私保护 | 照片仅在浏览器端处理，不上传至服务器存储 |
| 可用性 | 首屏加载 < 3秒，首次交互引导清晰 |

### 3.3 用户角色

本系统面向单类用户（无需区分角色）：网球或高尔夫的初学者及业余爱好者，希望通过AI技术检验和改善运动动作。

### 4.2 核心数据流

```
用户选择运动 & 动作类型
        │
        ▼
用户拍照 / 上传照片
        │
        ▼
MediaPipe Pose 提取关键点坐标 [x1,y1,z1, ... x33,y33,z33]
        │
        ▼
计算关节角度 + 绘制骨骼线叠加层
        │
        ▼
组装 Prompt（运动类型 + 动作类型 + 关节角度数据 + 偏差描述）
        │
        ▼
发送至后端 /api/analyze → 后端调用 DeepSeek API
        │
        ▼
返回：动作评价 + 偏差标注 + 训练步骤建议
        │
        ▼
前端渲染结果页（标注图 + AI报告 + 训练计划）
```

### 5.3 各模块详细说明

#### 模块1：运动类型选择器
- **功能**：提供网球🎾和高尔夫⛳两个入口
- **交互**：大卡片切换，选中态有视觉反馈
- **输出**：选中的运动类型（`tennis` / `golf`）

#### 模块2：动作类型选择器
- **功能**：根据运动类型展示可选动作
- **网球动作**：正手准备姿势、发球托举（Trophy Position）、半西方式握拍
- **高尔夫动作**：站位准备（Setup/Address）、上杆顶点（Top of Backswing）、推杆姿势
- **输出**：选中的动作类型ID

#### 模块3：拍照/上传模块
- **功能**：调用`getUserMedia`打开后置摄像头，或触发文件选择器
- **要求**：引导用户保持适当距离（全身入镜），提供参考线框辅助构图
- **输出**：包含完整人体的图片Base64/ImageData

#### 模块4：姿态检测与骨骼叠加
- **技术**：MediaPipe Pose（PoseLandmarker），模型 `lite` 版本适配移动端
- **处理**：输入图片 → 推理出33个地标点 → Canvas绘制关键点圆圈 + 骨骼连线
- **关键关节角度计算**：
  - 肩关节角度（大臂-躯干）
  - 肘关节角度（大臂-小臂）
  - 髋关节角度（躯干-大腿）
  - 膝关节角度（大腿-小腿）
  - 躯干前倾角

#### 模块5：AI分析报告模块
- **输入**：运动类型 + 动作类型 + 关节角度数据 + 偏差列表
- **Prompt设计**（内置运动知识库）：
  - 各动作的标准角度范围参考值
  - 常见错误模式及成因
  - 教练话术风格要求
- **DeepSeek API返回**：结构化JSON（摘要、问题列表、训练建议）
- **渲染**：问题高亮卡片 + 分步训练计划

#### 模块6：训练计划展示
- **功能**：将AI返回的训练建议渲染为带序号的分步卡片
- **每步包含**：训练名称、动作描述、建议组数/次数、注意事项

## 六、风险预估

| 编号 | 风险描述 | 影响等级 | 应对策略 |
|------|----------|----------|----------|
| R1 | MediaPipe在部分低端手机浏览器上性能不足 | 中 | 使用lite模型；检测超时(>5s)时提示用户使用更清晰的静态照片；备选：降级为服务端处理 |
| R2 | 用户拍摄照片不满足姿态检测要求（光线暗、身体不全、角度不佳） | 中 | 拍照前显示构图参考线框；检测失败时给出明确提示和重新拍摄引导 |
| R3 | DeepSeek API请求超时或限流 | 低 | 国内直连延迟低（<2s）；设置10秒超时；前端加载动画；失败提供重试按钮 |
| R4 | 网络环境差，首屏加载慢 | 低 | Vite代码分割；MediaPipe模型CDN加载；首屏不加载模型，进入拍照页再加载 |
| R5 | API Key泄漏 | 高 | Key仅存储在后端环境变量；前端不直接调用DeepSeek API；Vercel环境变量加密存储 |
| R6 | 开发时间不足 | 高 | 严格控制MVP范围；每日git commit追踪进度；第6天（6月17日）做功能冻结，剩余时间只做bug修复和打磨 |
| R7 | 不同设备摄像头兼容性问题 | 低 | 使用标准getUserMedia API；提供"从相册选择"作为备用方案 |
| R8 | 动作标准角度的"标准值"定义不精确 | 中 | 参考公开的运动生物力学文献；在Prompt中明确"标准范围为参考值，实际因人而异"；AI分析时考虑个体差异 |
